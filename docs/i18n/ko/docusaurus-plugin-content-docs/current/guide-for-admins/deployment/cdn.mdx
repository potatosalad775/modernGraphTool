# CDN 배포

modernGraphTool 애플리케이션 코드를 CDN (jsDelivr)에서 로드하고, 서버에는 설정, 테마, 데이터만 유지하는 방식입니다. 대부분의 운영자에게 권장되는 배포 모드입니다.

## 개요

CDN 모드에서는 서버가 최소한의 파일 세트만 호스팅합니다 — `index.html` 로더, `config.js`, `theme.css`, `data/`, `assets/`. 실제 애플리케이션 번들은 jsDelivr에서 제공됩니다. modernGraphTool의 새 버전이 릴리스되면 사이트가 자동으로 반영합니다: 파일 업로드도, 재배포도 필요하지 않습니다.

단점이라면, 사이트 로드 시 jsDelivr에 접근할 수 있어야 하고 *언제* 업데이트가 적용되는지를 직접 제어할 수 없다는 점입니다. 이 중 하나라도 문제라면 [사전 빌드 릴리스](./prebuilt)를 대신 사용하세요.

## 언제 선택해야 하나

**적합한 경우:**
- 최소한의 유지보수로 정상적인 사이트를 원하는 대부분의 운영자
- 통제된 릴리스 주기가 필요하지 않은 개인 데이터베이스
- 릴리스마다 수동 재업로드를 하고 싶지 않은 경우

**적합하지 않은 경우:**
- jsDelivr에 접근할 수 없는 에어갭 또는 인트라넷 배포
- 사용자가 받는 버전과 시점을 엄격히 제어해야 하는 사이트
- 앱 번들 자체를 깊이 커스터마이즈하는 경우 (포크, 커스텀 기능)

## 설치

1.  **다운로드** [최신 CDN 릴리스 zip](https://github.com/potatosalad775/modernGraphTool/releases) (이름이 `_cdn.zip`으로 끝나는 파일)을 내려받아 압축을 해제합니다.
2.  **`cdn-index.html` 복사** — 릴리스 (또는 소스의 `cdn/` 폴더)에서 파일을 서버로 복사하고 **`index.html`로 이름을 변경**합니다.
3.  서버에서 **다음 파일들을 그 옆에 함께 둡니다**:
    - `config.js` — 애플리케이션 설정
    - `theme.css` — 그래프 색상 테마
    - `data/` — 측정 데이터 및 타겟 커브
    - `assets/` — 이미지 및 문자열 파일
4.  **`config.js`에서 `CDN_MODE.BASE_PATH` 설정** — 사이트가 (서브)도메인의 루트에 위치하지 않는다면 반드시 설정해야 합니다. 이유는 [고급 섹션](#why-base_path-is-almost-always-required)을 참고하세요. `example.com/headphones/` 또는 `user.squig.link/headphones/`에 배포된 사이트의 경우:
    ```javascript
    CDN_MODE: {
      MAJOR_VERSION: 2,
      BASE_PATH: "/headphones",
    },
    ```
    마지막에 슬래시는 붙이지 않습니다. 사이트가 (서브)도메인의 루트에서 제공되는 경우 — 예: `example.com/` 또는 `user.squig.link/` — 는 `BASE_PATH`를 설정하지 않아도 됩니다. 같은 (서브)도메인에서 **여러 개의** modernGraphTool 인스턴스를 운영하는 경우 (squig.link에서 흔히 볼 수 있는 패턴 — 아래 팁 참고), 각 인스턴스는 자체 URL에 맞는 `BASE_PATH`가 설정된 자체 `config.js`를 가집니다.
5.  **(선택) `CDN_MODE.MAJOR_VERSION`으로 메이저 버전 고정**. 기본값은 항상 최신 메이저를 제공하는 것입니다.
6.  **업로드 후 테스트**. 페이지를 열고 기기를 로드한 뒤 공유 링크를 복사하여 새 브라우저 탭에서 열어 보세요. 공유 링크가 404라면 `BASE_PATH`를 다시 확인하세요 — 가장 흔한 설치 실수입니다.

여기까지 입니다. 고정한 메이저 버전 내에서 이후 릴리스는 자동으로 사이트에 반영됩니다.

## 버전 고정

기본적으로 CDN 로더는 가장 최신 메이저 버전으로 자동 업데이트됩니다. 사이트를 v2로 테스트했고 v3가 사전 공지 없이 적용되는 것을 원치 않는다면 메이저를 고정하세요:

```javascript
CDN_MODE: {
  MAJOR_VERSION: 2,
},
```

이렇게 하면 사이트는 v2.x.x에 머뭅니다 — v2 내에서의 마이너 및 패치 릴리스는 여전히 받지만, v3는 받지 않습니다. 올릴 준비가 되면 숫자를 변경하면 됩니다.

:::info[고정이 마이너 버전까지 막지는 않습니다]
`MAJOR_VERSION`은 메이저만 고정합니다. 고정된 메이저 내의 패치와 마이너 릴리스 (예: v2.1.0 → v2.2.0)는 여전히 자동으로 전개됩니다. 특정 패치 버전을 고정해야 한다면 대신 [사전 빌드 릴리스](./prebuilt)를 사용하세요.
:::

## 앱 업데이트

일반적인 운영에서는 할 일이 없습니다 — 고정된 메이저 내에서 업데이트가 자동으로 적용됩니다. 새 메이저로 올릴 준비가 되었다면:

1. `config.js`에서 `CDN_MODE.MAJOR_VERSION`을 올립니다 (예: `2` → `3`).
2. `config.js`를 다시 업로드합니다.
3. 사용자는 다음 페이지 로드에서 새 메이저를 받습니다.

그 사이 새로고침을 강제하고 싶은 사용자는 브라우저 캐시를 지우거나 하드 리로드를 수행하면 됩니다.

## 문제 해결

### 페이지가 404 오류와 함께 제대로 로드되지 않음

**원인:** `BASE_PATH`가 누락되었거나 잘못 설정됨.

**해결:** `config.js`를 열고 `CDN_MODE.BASE_PATH`를 하위 디렉토리 경로 (끝에 슬래시 없이)로 설정하세요. `example.com/headphones/`에 있다면 `"/headphones"`여야 합니다. 수정 후 캐시 없이 다시 로드하세요. 자세한 설명은 [왜 `BASE_PATH`는 거의 항상 필요한가](#why-base_path-is-almost-always-required)를 참고하세요.

### 새 릴리스가 나왔는데도 오래된 버전에 머물러 있음

**원인:** 브라우저 캐시, jsDelivr 엣지 캐시, 또는 잊고 있던 `MAJOR_VERSION` 고정.

**해결:**
1. `CDN_MODE.MAJOR_VERSION`이 원하는 것보다 낮은 메이저로 고정되어 있지 않은지 확인하세요.
2. 브라우저 캐시를 지우거나 하드 리로드를 수행합니다 (Ctrl+Shift+R / Cmd+Shift+R).
3. `versions.json` 이외의 파일은 jsDelivr 엣지 캐시를 기다려야 합니다 — POP에 따라 몇 분에서 몇 시간 내에 클리어됩니다.

### 앱이 전혀 로드되지 않음

**원인:** 네트워크 접근성, 설정 구문 오류, 또는 해당 지역에서 CDN 차단.

**해결:**
1. 브라우저 네트워크 탭을 열어 어떤 요청이 실패하고 있는지 확인하세요.
2. `versions.json` 또는 `boot.json` 요청이 실패한다면 해당 위치에서 jsDelivr / GitHub raw에 접근할 수 없다는 뜻입니다. 대안으로 [사전 빌드 릴리스](./prebuilt)를 고려하세요.
3. 아무것도 로드되지 않고 네트워크 활동도 없다면 `config.js`에 구문 오류가 있을 수 있습니다 — 브라우저 콘솔을 확인하세요.

### 개발에서는 되는데 프로덕션에서는 깨짐

**원인:** 거의 항상 `BASE_PATH`입니다. 로컬 미리보기는 보통 `localhost:5173/`에서 실행되어 사실상 루트에 있지만, 프로덕션 서버는 하위 디렉토리에 있습니다.

**해결:** 프로덕션 하위 디렉토리에 맞춰 `BASE_PATH`를 설정하세요.

### `versions.json: HTTP 404`

**원인:** `BASE`가 `versions.json` 파일이 없는 위치를 가리키고, 명시적인 `VERSIONS_URL` 오버라이드도 없음.

**해결:** `BASE` 루트에 `versions.json`을 추가하거나, `VERSIONS_URL`을 올바른 URL로 명시적으로 설정하세요.

## 고급

### 업데이트가 전파되는 방식

메이저 버전을 특정 릴리스 번호로 매핑하는 파일인 `versions.json`은 jsDelivr가 **아닌** `raw.githubusercontent.com`에서 직접 가져옵니다. 이는 의도된 것입니다: jsDelivr의 엣지 캐시는 오래된 응답을 몇 시간 동안 고정할 수 있으며, `versions.json`은 그 지연이 중요한 유일한 파일입니다. GitHub raw에서 가져오면 캐시를 우회해 새 릴리스가 게시된 뒤 몇 분 이내에 사용자에게 도달할 수 있습니다.

### `VERSIONS_URL` 재정의

`VERSIONS_URL`은 기본적으로 `BASE`에서 유도된 `raw.githubusercontent.com` URL로 설정됩니다. `BASE`가 인식 가능한 jsDelivr `gh/` URL이 아닌 경우 (예를 들어 직접 호스팅하는 경우), 유도 결과는 `null`이 되고 로더는 `${BASE}/versions.json`으로 폴백합니다. 둘 다 적합하지 않은 경우에는 `VERSIONS_URL`을 명시적으로 재정의하세요. 로더가 기대하는 URL 형태는 [cdn/loader.js:186-196](https://github.com/potatosalad775/modernGraphTool/blob/main/cdn/loader.js#L186-L196)에 문서화되어 있습니다.

*다른* 모든 파일 (앱 청크, CSS, preload)은 일반적인 캐시 동작을 따라 jsDelivr에서 제공되며, 엣지 위치에 따라 몇 분에서 몇 시간이 걸립니다. 가장 최신 버전으로 강제 새로고침하고 싶은 사용자는 브라우저 캐시를 지우거나 하드 리로드를 수행할 수 있습니다.

### CDN 자산 직접 호스팅

jsDelivr를 사용할 수 없는 경우 — 프라이빗 포크, 에어갭 미러, 대역폭 문제 등 — CDN 자산을 직접 호스팅하고 `CDN_MODE.BASE`를 자신의 URL로 지정할 수 있습니다:

```javascript
CDN_MODE: {
  MAJOR_VERSION: 2,
  BASE: "https://my-mirror.example.com/moderngraphtool",
  VERSIONS_URL: "https://my-mirror.example.com/moderngraphtool/versions.json",
},
```

전체 구조를 미러링해야 합니다: 루트의 `versions.json` + `v{version}/boot.json` + 참조되는 모든 청크. 기본 GitHub raw 유도는 `BASE`가 jsDelivr `gh/` URL을 가리킬 때만 동작하므로 `VERSIONS_URL`을 `BASE`와 함께 지정해야 합니다.

:::warning[트레이드오프]
CDN 자산을 직접 호스팅하면 CDN 모드의 "수동 업데이트 없음" 이라는 장점이 사라집니다. 이제 사이트 *와* 미러라는 두 가지를 유지보수해야 합니다. 대부분의 운영자에게는 [사전 빌드 릴리스](./prebuilt) 경로가 더 단순합니다.
:::

### 왜 `BASE_PATH`는 거의 항상 필요한가 {#why-base_path-is-almost-always-required}

로더는 URL에서 배포 경로를 자동으로 감지하려고 하지만, 자동 감지는 도메인 루트 배포에서만 동작합니다. 이유는 다음과 같습니다.

방문자가 `https://example.com/headphones/`를 열 때, 브라우저의 `window.location.pathname`은 `/headphones/`입니다 — `/headphones/index.html`이 **아닙니다**. 서버가 기본값으로 `index.html`을 제공하더라도 URL 바에는 디렉토리가 표시됩니다. 로더는 pathname이 `.html`로 끝나는지 확인하고, 그렇지 않다면 앱이 도메인 루트에 호스팅되어 있다고 가정하고 빈 base path를 사용합니다. 이는 `example.com/`에는 맞지만 `example.com/headphones/`에는 틀리며, `example.com/headphones/share/abc` 같은 모든 공유 링크 딥 라우트에도 틀립니다.

실제 결과:

| 배포 | 자동 감지된 base | 정확한가? | 조치 |
|---|---|---|---|
| `example.com/` | `""` | 맞음 | `BASE_PATH`를 설정하지 않음 |
| `example.com/headphones/` | `""` | **아님 — `/headphones`여야 함** | `BASE_PATH: "/headphones"` 설정 |
| `example.com/headphones/share/abc` | `""` | **아님 — `/headphones`여야 함** | `BASE_PATH: "/headphones"` 설정 |
| `example.com/headphones/cdn-index.html` | `/headphones` | 맞음 | (하지만 아무도 이 URL을 방문하지 않습니다 — 설치 가이드는 파일 이름을 변경하도록 안내합니다.) |

요약: **사이트가 도메인 루트에 있지 않다면 `BASE_PATH`를 설정하세요.** 이를 빠뜨렸을 때의 전형적인 증상은, 홈 URL을 방문하면 페이지가 정상 로드되지만 공유 링크가 모두 404가 되는 것입니다. 정확한 감지 규칙을 읽어보고 싶다면 로더 로직이 [cdn/loader.js:111-137](https://github.com/potatosalad775/modernGraphTool/blob/main/cdn/loader.js#L111-L137)에 있습니다.

:::tip[squig.link 운영자]
각 squig.link 사용자는 자신만의 서브도메인 — `<username>.squig.link` — 을 받고, 루트와 그 아래에 만드는 모든 하위 디렉토리에 완전한 접근 권한을 가집니다. 흔한 패턴은 같은 서브도메인 위에 **두 개의** modernGraphTool 인스턴스를 운영하는 것입니다:

- `<username>.squig.link/` — 보통 이어폰 데이터베이스. 서브도메인 루트에 있으므로 이 인스턴스는 **`BASE_PATH`를 설정하지 않습니다**.
- `<username>.squig.link/headphones/` — 하위 디렉토리의 헤드폰 데이터베이스. 이 인스턴스는 자체 `config.js`에 **`BASE_PATH: "/headphones"`**를 설정해야 합니다.

각 인스턴스는 자체 `index.html`, 자체 `config.js`, 자체 `data/` 등을 가집니다. `BASE_PATH`는 인스턴스별로 적용되므로 `/headphones/` 인스턴스에만 설정하면 됩니다. 추가 하위 디렉토리 (예: `/iems/`, `/speakers/`)를 만드는 경우에도 각각 경로에 맞는 `BASE_PATH`가 필요합니다.
:::
