# 소스에서 빌드

modernGraphTool 저장소를 클론하고, 원하는 대로 커스터마이즈한 뒤, 자신만의 `dist/` 폴더를 빌드하는 방식입니다. 이 경로는 `config.js`에서 노출하는 범위를 넘어서는 변경이 필요한 개발자와 운영자를 위한 것입니다.

## 개요

소스에서 빌드하면 앱 번들 자체를 완전히 제어할 수 있습니다 — 컴포넌트를 수정하고, 기능을 추가하고, 빌드 파이프라인을 변경하고, 커스텀 릴리스를 배포할 수 있습니다. 그만큼 가장 많은 작업이 필요합니다: Node.js, `git`, 그리고 빌드 오류를 디버깅할 수 있는 능력이 필요합니다.

단지 `config.js`, `theme.css`, `data/`, `assets/`만 커스터마이즈하고 싶다면 소스에서 빌드할 필요가 없습니다 — 대신 [CDN 배포](./cdn) 또는 [사전 빌드 릴리스](./prebuilt)를 사용하세요.

## 언제 선택해야 하나

**적합한 경우:**
- 설정 표면이 노출하는 범위를 넘어서는 커스텀 기능이 있는 포크
- 업스트림에 기여하고자 하는 개발자
- 앱 컴포넌트, 라우트, 또는 빌드 설정을 수정해야 하는 운영자

**적합하지 않은 경우:**
- 단지 정상적인 사이트를 원하는 운영자 — CDN 또는 사전 빌드를 사용하세요.
- `node_modules`와 빌드 오류를 다루고 싶지 않은 경우

## 사전 요구 사항

- [Node.js](https://nodejs.org/) v18 이상
- `git`
- 기본적인 커맨드라인 사용법

## 클론 및 설치

```bash
git clone https://github.com/potatosalad775/modernGraphTool.git
cd modernGraphTool
npm install
```

## `defaults/` vs `static/`

modernGraphTool은 2계층 정적 자산 시스템을 사용합니다. 편집을 시작하기 전에 이를 이해하면 많은 시간을 아낄 수 있습니다.

- **`defaults/`** — 사용자가 설정할 수 있는 템플릿을 담습니다: `config.js`, `theme.css`, `data/`, `assets/` 등. 일반 운영자가 편집하게 되는 파일들입니다. git에 체크인되어 있으며 모든 릴리스와 함께 제공됩니다.
- **`static/`** — gitignore된 오버라이드 레이어입니다. `static/`에 같은 경로로 배치한 파일은 `defaults/`의 동일 경로 파일보다 우선합니다 — `npm run dev`와 `npm run build` 모두에서.

**개발 중 커스터마이즈 패턴:** 편집하려는 파일을 `defaults/`에서 `static/`으로 복사한 뒤 `static/`의 사본을 수정하세요. 이렇게 하면 개인 커스터마이징이 git에서 분리되고 `git pull`이 매끄러워집니다.

폴백 동작은 [vite-plugin-defaults.ts](https://github.com/potatosalad775/modernGraphTool/blob/main/vite-plugin-defaults.ts)에 구현되어 있으며, dev 중에는 `defaults/`를 폴백으로 제공하고 빌드 후에는 `static/`의 파일을 덮어쓰지 않으면서 `dist/`로 복사합니다.

:::warning[`defaults/`나 `static/`에서 모듈로 import하지 마세요]
`defaults/`와 `static/`의 파일들은 런타임에 로드되는 자산이지 번들된 모듈이 아닙니다. 소스에서 `import`하지 말고, 런타임에 `fetch()`로 읽어오세요.
:::

## 로컬 개발

```bash
npm run dev
```

Vite dev 서버가 `http://localhost:5173`에서 HMR과 함께 시작됩니다. `.svelte`, `.ts`, `.css`, 그리고 `defaults/` 파일 편집 모두 실시간으로 반영됩니다.

자신의 측정 데이터로 테스트하고 싶다면 `data/` 폴더를 `static/data/`에 두세요 — dev 서버는 기본 파일 대신 사용자의 파일을 제공합니다.

## 프로덕션 빌드 미리 보기

```bash
npm run build
npm run preview
```

`npm run build`는 `dist/`를 생성하고 `npm run preview`는 이 폴더를 로컬에서 제공합니다. 업로드 전 프로덕션 번들을 검증하는 방법입니다. 미리 보기 서버는 기본적으로 `http://localhost:4173`에서 실행됩니다.

## 타입 체크 및 품질 게이트

```bash
npm run check          # TypeScript + Svelte 타입 체크
npm run lint           # Prettier + ESLint
npm run test           # Vitest 단위 테스트
npm run format         # 포매팅 자동 수정
```

커스텀 빌드를 배포하기 전에 `npm run check`와 `npm run test`를 실행하세요 — 대부분의 통합 문제는 사용자에게 도달하기 전에 잡힙니다.

## 빌드 및 배포

```bash
npm run build
```

빌드 결과물은 `dist/`에 생성됩니다. 여기서부터의 배포는 [사전 빌드 릴리스](./prebuilt)와 동일합니다 — 폴더를 웹 서버에 업로드하고 SPA 폴백 (Apache는 `.htaccess`, 기타 서버는 동등한 설정)이 구성되어 있는지 확인하세요.

자체 미러를 위해 CDN 모드 빌드도 생성하고 싶다면 다음을 실행하세요:

```bash
npm run build:cdn
```

이 명령은 CDN 로더가 기대하는 버전별 번들 구조가 포함된 `dist-cdn/` 폴더를 생성합니다.

## 업스트림 업데이트 받기

```bash
git pull
npm install
npm run build
```

`defaults/`의 파일을 (복사해서 `static/`에 두는 대신) 직접 편집해 왔다면 업스트림 업데이트에서 병합 충돌이 발생할 수 있습니다. 깔끔한 해결책은 한 번만 `static/`으로 커스터마이즈를 옮기는 것입니다 — 그러면 `git pull`이 사용자의 편집과 영원히 충돌하지 않습니다.

## 문제 해결

### TypeScript 오류로 빌드 실패

**원인:** 보통 `git pull` 이후 `.svelte-kit/` 디렉토리가 오래되었기 때문입니다.

**해결:** `npm run prepare` (내부적으로 `svelte-kit sync`를 실행)를 실행한 뒤 `npm run build`를 재시도하세요. 여전히 실패한다면 `node_modules/`와 `.svelte-kit/`를 삭제하고 다시 설치하세요.

### `npm run dev`가 포트 충돌로 실패

**원인:** 포트 5173이 이미 사용 중입니다 (다른 Vite 인스턴스 또는 다른 dev 서버).

**해결:** 다른 프로세스를 중지하거나 Vite에 `--port`를 전달하세요: `npm run dev -- --port 5174`.

### `defaults/config.js`를 수정했는데 반영되지 않음

**원인:** `static/config.js`가 그것을 가리고 있습니다.

**해결:** 대신 `static/config.js`를 편집하거나, `defaults/`가 우선하도록 하려면 삭제하세요.

### 새로 클론한 상태에서 테스트 실패

**원인:** Paraglide i18n 코드 생성 단계가 빠짐.

**해결:** 테스트 실행 전에 `npm run prepare`를 실행하세요 — 이는 `svelte-kit sync`를 실행하며, 그 과정에서 Paraglide 컴파일 단계를 트리거합니다.
