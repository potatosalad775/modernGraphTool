# v1에서 v2로 마이그레이션

이 페이지는 기존 modernGraphTool v1 데이터베이스를 v2로 옮기기 위한 단계별 체크리스트입니다.

:::tip[하이라이트를 찾으시나요?]
어떤 점이 달라졌는지 기능 단위로 먼저 확인하고 싶다면 [v2의 새로운 기능](./whats-new-in-v2) 페이지를 참고하세요. 이 페이지는 마이그레이션 절차에만 집중합니다.
:::

## 호환성

**기존 데이터는 그대로 사용할 수 있습니다.** 폴더 구조와 파일 형식은 변경되지 않았습니다:

- `phone_book.json` — 동일한 구조, 동일한 위치 (`data/phone_book.json`)
- FR 측정 파일 — `data/phones/`에서 동일한 `.txt` 형식을 사용합니다.
- 타겟 커브 파일 — `data/target/`에서 동일한 `.txt` 형식을 사용합니다.
- Preference Bound 파일 — `Bounds U.txt` / `Bounds D.txt`는 여전히 `.txt`이지만 위치가 변경되었습니다 (아래 [마이그레이션 단계](#migration-steps)를 참고하세요 — v1의 `extensions/preference-bound/data/`에서 `data/`로 이동).

`config.js`는 전반적인 구조가 동일하게 유지됩니다. v1의 모든 설정 (`INITIAL_PHONES`, `INITIAL_TARGETS`, `NORMALIZATION`, `VISUALIZATION`, `INTERFACE`, `PATH`, `WATERMARK`, `TARGET_MANIFEST` 등)은 그대로 호환됩니다. v2는 몇 가지 선택적인 새 섹션을 추가했습니다 — [v2의 새로운 기능](./whats-new-in-v2#new-config-sections)을 참고하세요.

배포 방식도 동일합니다 — v2는 완전한 정적 SPA를 `dist/`로 출력합니다. 해당 폴더를 웹 서버에 복사하세요. Apache용 SPA fallback을 위한 `.htaccess` 파일도 함께 포함되어 있습니다.

## 마이그레이션 단계 {#migration-steps}

1. **다운로드** [최신 v2 릴리스](https://github.com/potatosalad775/modernGraphTool/releases)를 내려받습니다.
2. **데이터 복사** — 기존의 `data/` 폴더 (phones, targets, `phone_book.json`)를 v2 릴리스 폴더로 옮깁니다.
3. **Preference Bound 데이터 파일 이동** — v1에서 `Bounds U.txt`, `Bounds D.txt`는 `extensions/preference-bound/data/`에 있었습니다. v2에서는 이 파일들이 `data/` 바로 아래, `phones/`와 `target/`과 나란히 있어야 합니다. 이 단계를 건너뛰면 Preference Bound 오버레이를 활성화해도 그래프에 표시되지 않습니다. 기준 DF 타겟 파일 (예: `KEMAR DF (KB006x) Target.txt`)은 2단계에서 이미 `data/target/`에 있어야 합니다.
4. **`config.js` 마이그레이션** — v1의 `config.js` 설정을 v2 `config.js` 템플릿에 복사합니다. 핵심 설정은 동일하며, [새 섹션](./whats-new-in-v2#new-config-sections) (`SQUIGLINK`, `TARGET_CUSTOMIZER`, `PREFERENCE_BOUND`, `MULTI_SAMPLE`, `HPTF`, `CDN_MODE`)을 검토하고 필요에 맞게 조정하세요. [Config Editor](/config-generator)를 사용하면 모든 설정을 손쉽게 마이그레이션하고 검토할 수 있습니다.
5. **`theme.css` 마이그레이션** — `theme.css`를 커스터마이즈했다면 색상 값을 v2 형식(그래프 스타일링용 CSS 커스텀 속성 + OKLCH 기반 UI 변수)으로 이식해야 합니다. [테마 생성기](/theme-generator)를 사용하면 시각적으로 생성할 수 있습니다.
6. **페이지 메타데이터 마이그레이션 (`index.html`)** — v2는 자체 `index.html`을 제공하므로 v1 `index.html`에 가한 수정 사항은 자동으로 이전되지 **않습니다**. v2의 `index.html`에 다시 적용하세요:
   - `<title>` 및 `<meta name="title">`
   - `<meta name="description">` 및 `<meta name="keywords">`
   - Open Graph 태그: `<meta property="og:title">`, `<meta property="og:description">`, `<meta property="og:url">`, `<meta property="og:image">`
   - `<link rel="canonical">`
   - 파비콘: `<link rel="icon">`, `<link rel="shortcut icon">`, `<link rel="apple-touch-icon">`
   - `<meta name="apple-mobile-web-app-title">`
   - 웹 앱 매니페스트: `<link rel="manifest">` (사용 중인 경우)

   필드별 상세 내용은 [페이지 커스터마이징 → 페이지 메타데이터 변경](./guide-for-admins/customize-page#changing-page-metadata-indexhtml) 섹션에서 확인할 수 있습니다.

   :::info[소스 빌드 사용자]
   소스에서 빌드하는 경우 `index.html` 대신 `src/app.html`을 수정하세요. 빌드 결과인 `dist/index.html`은 이 템플릿으로부터 생성됩니다.
   :::
7. **Extension 삭제** — `extensions/` 폴더는 더 이상 사용되지 않습니다. extension 관련 설정은 이제 `config.js`에 있어야 합니다.
8. **로컬 테스트** — 브라우저에서 페이지를 열어 데이터가 올바르게 로드되는지 확인합니다.

## 배포 옵션 선택

v1은 사전 빌드 방식으로만 제공되었습니다 — 릴리스를 내려받고, 업로드하면 끝이었습니다. v2는 여기에 두 가지 옵션이 추가되었습니다:

- **[사전 빌드 릴리스](./guide-for-admins/deployment/prebuilt)** — v1 설정을 그대로 대체할 수 있는 방식입니다. 릴리스를 다운로드하고, 데이터와 설정을 복사한 뒤 업로드하면 됩니다. 설정해야 할 `BASE_PATH`가 없으며, v1과 동일한 방식으로 동작합니다.
- **[CDN 배포](./guide-for-admins/deployment/cdn)** — 권장 방식이며, 서버에는 `index.html`, `config.js`, `theme.css`, `data/`, `assets/`만 두고 앱 번들은 jsDelivr에서 로드되어 자동으로 업데이트됩니다.
- **[소스에서 빌드](./guide-for-admins/deployment/from-source)** — 개발자 및 포크를 위한 옵션입니다.

:::tip[CDN 대안]
사전 빌드된 릴리스를 전체적으로 호스팅하는 대신 [CDN 배포 모드](./guide-for-admins/deployment/cdn) 사용을 고려해 보세요. 앱 코드를 직접 관리할 필요가 없고, 향후 업데이트도 자동으로 적용됩니다.
:::

### CDN 모드를 선택했다면 `BASE_PATH`를 꼭 확인하세요

v2 CDN 모드에서는 (서브)도메인의 루트가 **아닌** 경로에 배포할 경우, `config.js`에 `CDN_MODE.BASE_PATH`가 반드시 필요합니다. 설정하지 않으면 홈 페이지는 정상적으로 로드되지만 모든 공유 링크가 404가 됩니다. 수정은 설정 한 줄이면 됩니다. `example.com/headphones/`에 배포된 사이트의 경우:

```javascript
CDN_MODE: {
  MAJOR_VERSION: 2,
  BASE_PATH: "/headphones",
},
```

상세한 설명과 예시는 [왜 `BASE_PATH`는 거의 항상 필요한가](./guide-for-admins/deployment/cdn#why-base_path-is-almost-always-required)를 참고하세요.

**squig.link 운영자**는 보통 같은 서브도메인 아래에 두 개의 인스턴스를 운영합니다: `<username>.squig.link/`는 이어폰용 (서브도메인 루트에 위치 — `BASE_PATH` 불필요), `<username>.squig.link/headphones/`는 헤드폰용 (`BASE_PATH: "/headphones"` 필요)입니다. 각 인스턴스는 자체 `config.js`를 가지므로 `/headphones/` 인스턴스에만 `BASE_PATH`를 설정하면 됩니다. 루트의 이어폰 인스턴스는 그대로 두면 됩니다.

**개인 데이터베이스**를 `example.com/headphones/`와 같은 하위 디렉토리에 호스팅하는 경우에도 `BASE_PATH`를 맞춰주어야 합니다. `example.com/` (도메인 루트)에 배포된 데이터베이스는 필요하지 않습니다.

### 이후 CDN과 사전 빌드를 오가려면

앱 번들을 제외한 모든 것 (데이터, 설정, 테마, Preference Bound 파일)은 두 옵션 사이에서 동일합니다. `index.html` 파일을 교체하고 필요에 따라 `config.js`의 `CDN_MODE` 섹션을 추가하거나 제거하는 것만으로 두 방식 사이를 오갈 수 있습니다. 잠금(lock-in)은 없습니다.
